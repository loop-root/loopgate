ReadMe — Governed execution

Local-first control plane

Governed AI execution, without treating the model as authority.

Loopgate is the enforcement layer for real work with agents: capabilities, approvals, policy, secrets, sandbox mediation, append-only audit, continuity memory, and morphling lifecycle—so natural language never mints permission on its own.

What Loopgate is

A policy-governed runtime that sits between your operator client (for example, Haven) and anything privileged: filesystem tools, integrations, and durable memory writes. It is not a reverse proxy—it is where allow/deny actually happens.

  • Capabilities & policy

    Deny-by-default evaluation, scoped capability tokens, and typed results—so only explicitly allowed actions run.

  • Approvals & audit

    Human-in-the-loop decisions enforced in the control plane, with append-only, hash-linked event history for security-relevant transitions.

  • Secrets & integrations

    Provider credentials stay inside Loopgate. The client sees redacted metadata and structured outcomes—not raw tokens.

  • Sandbox mediation

    Imports, staging, inspection, and export move through explicit, mediated steps under /morph/home—not ad hoc host access from the model.

  • Morphlings

    Bounded subordinate workers with a Loopgate-owned lifecycle: spawn, run, complete, review, terminate—never self-authorizing background agents.

  • Memory & continuity

    Governed recall, inspection, and explicit writes—continuity is bounded and policy-shaped, not an immutable dump of raw chat.

How the pieces fit

Three roles keep authority clear: the operator supplies intent and approval; the client handles conversation and session UX; Loopgate owns tokens, policy, execution, and authoritative control-plane audit.

  1. Operator

    You ask for real work in plain language—for example: “Can you check my calendar and email Karen to line up an appointment?” You also review approvals when Loopgate says a step needs an explicit yes or no.

  2. Client

    The app you’re already in: Haven, or a coding assistant such as Claude Code, Cursor, Codex, Antigravity, and similar tools. It runs the chat, shows tool suggestions, and renders Loopgate’s allow/deny outcomes—it does not get to bypass them.

  3. Loopgate

    The control plane that decides what is allowed. It evaluates policy, mints capability tokens, holds secrets, mediates sandbox and integrations, governs morphlings and memory writes, and records authoritative audit—every privileged step goes through it or it doesn’t happen.

Core rule: the model never calls third-party systems directly. The path is model output → client validation → Loopgate capability request → mediated execution → structured response.

Design pillars

  • Authority Natural language is not permission—only the control plane mints and spends capability.
  • Untrusted by default Model output, tool output, and files are content until validated by policy and types.
  • Local v1 Supported transport is HTTP on a Unix domain socket on the same machine—not a public network API.
  • Enterprise direction MCP server, transparent proxy, multi-tenant tenant_id isolation, and admin tooling—without relaxing the same policy and audit rules as local handlers.

Status

Loopgate is experimental and under active hardening. Treat it as not production-ready security software without your own review. The implementation is proprietary—see the LICENSE in the repository.

Technical documentation, architecture, setup, and threat modeling live alongside the code in the docs/ tree (start with docs/README.md and docs/design_overview/).
