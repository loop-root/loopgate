# RFC-MORPH-0004: Sandbox filesystem policy (Loopgate-enforced)

- **Status:** Draft — target `/morph/home` product boundary vs current repo-root layouts
- **Primary authority:** **Loopgate** (policy + mediation)
- **Normative revision:** 2026-03-09

---

# Summary

This RFC defines the **sandbox filesystem model** used by **Haven** (operator UI) and **Morphlings**. The sandbox is a controlled workspace that prevents agents from roaming the host system while still allowing useful work to occur.

All Morph and Morphling filesystem activity must occur inside a designated sandbox root:

```
/morph/home
```

Everything outside this path is owned and controlled by **Loopgate**.

This is the target product boundary. Current code already enforces
symlink-safe, fail-closed access inside configured roots, but the checked-in
runtime policy is still repo-root based rather than `/morph/home`-based.

The sandbox model follows a strict workflow:

```
Import → Work → Review → Export
```

This ensures that agents cannot silently modify host files and that all changes are visible and auditable.

---

# Design Goals

The sandbox must provide:

- strict filesystem boundaries
- deterministic path validation
- clear provenance for all files
- explicit import/export operations
- protection against filesystem escape attacks

The sandbox must never allow:

- root filesystem traversal
- unrestricted home directory access
- implicit file discovery outside sandbox

---

# Filesystem Root

Loopgate owns the top-level Morph directory:

```
/morph
```

Morph and Morphlings may only operate inside:

```
/morph/home
```

Example structure:

```
/morph

  /config
  /policy
  /state
  /var

  /home
```

Everything above `/morph/home` is considered **Loopgate control-plane territory**.

Morph must not mutate files outside `/morph/home`.

---

# Sandbox Directory Layout

Inside `/morph/home` the workspace is structured into functional zones:

```
/morph/home

workspace/
imports/
outputs/
scratch/
agents/
quarantine/
tmp/
logs/
```

---

# Filesystem Zones

## Workspace

Primary working directory for active tasks.

Example:

```
/morph/home/workspace/project-x
```

Morphlings may read and write here depending on their capability token.

---

## Imports

Imported files from the host system.

Example:

```
/morph/home/imports/project-x
```

Files here should be treated as **snapshots** of the original source.

Agents should not assume these are the live host files.

---

## Outputs

All artifacts generated by Morphlings must be written here.

Example artifacts:

- patch files
- generated code
- reports
- analysis output

Artifacts must not be exported automatically.

---

## Scratch

Temporary workspace for intermediate files.

Scratch contents may be deleted automatically.

---

## Agents

Each morphling receives its own working directory.

Example:

```
/morph/home/agents/task-uuid
```

This directory is created by Loopgate at morphling spawn time.

---

## Quarantine

Untrusted external content must be stored here.

Examples:

- downloaded web content
- provider responses
- extracted documents

Promotion from quarantine must require explicit validation.

---

## tmp

Ephemeral runtime files.

Contents may be deleted at any time.

---

## logs

Human-readable logs and debugging output.

Audit logs remain outside the sandbox under Loopgate control.

---

# Import Workflow

Files must be **explicitly imported** into the sandbox.

Example:

```
import ~/repo/project-x → /morph/home/imports/project-x
```

Import behavior:

- files are copied or snapshotted
- original host files remain unchanged

Import operations must be mediated by Loopgate.

Imports must not be implemented as host symlinks into the sandbox.

The safe default is:

- copy
- snapshot
- clone/reflink where platform-safe and policy-approved

but not symbolic-link passthrough to host files.

---

# Export Workflow

Files inside the sandbox must not modify host files directly.

Exports require approval.

Example:

```
/morph/home/outputs/patch.diff
→ export to ~/repo/project-x
```

Export process:

1. Morph stages artifact
2. Morph summarizes change
3. User approves export
4. Loopgate applies change

---

# Path Validation

All filesystem operations must be validated by Loopgate.

Validation steps:

1. canonicalize requested path
2. resolve symlinks
3. verify path remains inside `/morph/home`
4. verify capability token allows access

Requests failing validation must be rejected.

---

# Symlink Handling

Symlinks may exist but must be resolved before use.

Loopgate must:

1. resolve the symlink
2. canonicalize the resulting path
3. verify the resolved path remains inside `/morph/home`

If the resolved path escapes the sandbox, access must be denied.

Symlinks should resolve only when the **final canonical target remains inside
the sandbox root**. A symlink is storage indirection, not a permission
expansion mechanism.

This prevents **filesystem escape attacks**.

---

# Provenance Tracking

Imported files should include provenance metadata.

Example:

```
import_manifest.json

source_path: ~/repo/project-x
snapshot_time: 2026-03-09
hashes:
  fileA: sha256...
  fileB: sha256...
```

This allows traceability for all sandbox inputs.

---

# Disk Limits

Loopgate should enforce sandbox resource limits.

Recommended constraints:

- maximum workspace size
- per-morphling disk quota
- automatic cleanup of scratch/tmp

---

# Security Invariants

The sandbox must guarantee:

1. Agents cannot access files outside `/morph/home`.
2. Agents cannot modify Loopgate control-plane files.
3. All imports are explicit.
4. All exports require approval.
5. Symlinks cannot escape sandbox boundaries.
6. Quarantined content remains untrusted until promoted.

---

# Future Work

Potential improvements:

- snapshot-based sandboxing
- overlay filesystems
- containerized morphling execution
- filesystem rollback

These features are **out of scope for v1**.

---

# Conclusion

The sandbox filesystem ensures that Morph and Morphlings operate within a controlled environment. By enforcing strict import/export rules and isolating execution within `/morph/home`, the system prevents unintended modification of the host filesystem while maintaining a safe and auditable workspace.
