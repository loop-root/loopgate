# RFC-MORPH-0003: Loopgate capability token model

- **Status:** Draft — aligned to Loopgate token semantics (see also `docs/rfcs/0001-loopgate-token-policy.md`)
- **Primary authority:** **Loopgate**
- **Normative revision:** 2026-03-09

---

# Summary

Loopgate issues **capability tokens** that grant narrowly scoped authority to **operator clients** and **morphlings**. These tokens act as the system's equivalent of **kernel-issued process permissions**, ensuring that no execution context can perform actions outside explicitly authorized bounds.

Capability tokens are the primary enforcement mechanism that separates **planning (operator client)** from **authority (Loopgate)** and **execution (Morphlings)**.

No operation that mutates state, accesses external resources, or performs privileged filesystem activity may occur without a valid capability token issued by Loopgate.

---

# Design Goals

The token model must provide:

- strict scope enforcement
- deterministic validation
- replay protection
- time-bound permissions
- audit traceability
- minimal authority delegation

Tokens must always grant the **minimum authority required for a task**.

---

# Capability Token Structure

Capability tokens are issued by Loopgate and validated by Loopgate.

For v1/v2 local control-plane behavior, capability tokens should be treated as
**opaque scoped credentials**, not as self-validating JWT-like documents.
Authority comes from:

- server-side token records
- task/session binding
- path/capability scope
- expiration
- signed request envelopes
- replay protection

The JSON structure below is a **conceptual scope model**, not a requirement
that the token itself be directly inspectable by operator clients or morphlings.

Example structure:

```json
{
  "token_id": "uuid",
  "issued_by": "loopgate",
  "issued_to": "morphling-uuid",
  "task_id": "uuid",
  "capabilities": [
    "read_path",
    "write_path",
    "propose_patch"
  ],
  "allowed_paths": [
    "/morph/home/agents/task-uuid",
    "/morph/home/workspace/project-x"
  ],
  "expiration": "2026-03-09T22:00:00Z",
  "issued_at": "2026-03-09T21:55:00Z",
  "nonce": "random-value",
  "signature": "loopgate-signature"
}
```

---

# Token Fields

## token_id

Globally unique identifier for the capability token.

## issued_by

Authority issuing the token. Must always be `loopgate`.

## issued_to

Identity receiving the token.

Examples:

- morph
- morphling-uuid

## task_id

Identifier linking the token to a specific morphling task.

## capabilities

List of explicit permissions granted to the token holder.

Examples:

- read_path
- write_path
- propose_patch
- execute_test
- provider_query
- export_artifact

Capabilities must never be implied.

## allowed_paths

Filesystem paths the token holder may access.

Access outside these paths must be denied.

## expiration

Timestamp when the token becomes invalid.

## issued_at

Timestamp when token was created.

## nonce

Unique random value used to prevent replay attacks.

## signature

Conceptual representation of Loopgate-issued authority.

In current architecture, request integrity is enforced primarily through the
Loopgate signed request envelope and server-side token validation rather than a
self-contained signed token payload.

---

# Token Issuance Flow

Tokens are issued only after Loopgate validates a request.

```
Operator client planning
→ Operator client requests capability
→ Loopgate evaluates policy
→ Loopgate requests approval if required
→ Loopgate issues scoped token
→ Operator client spawns morphling with token
```

Tokens must never be self-generated by operator clients or morphlings.

---

# Token Validation

Before performing any operation, Loopgate must validate:

1. token validity in Loopgate-controlled state
2. token expiration
3. request nonce / replay protection
4. capability inclusion
5. filesystem path scope
6. task binding

Failure of any validation must deny the operation.

---

# Token Scope Rules

Capability tokens must enforce the principle of **least privilege**.

Tokens must be restricted by:

- task
- morphling identity
- capability type
- filesystem scope
- expiration time

Tokens must not grant:

- unrestricted filesystem access
- access outside `/morph/home`
- policy mutation
- audit log mutation
- provider secret access

---

# Token Expiration

Tokens must expire quickly.

Recommended defaults:

| Context | Expiration |
|--------|------------|
| Operator client planning | short-lived |
| Morphling execution | task duration |
| Export approval | immediate execution |

Expired tokens must be rejected.

---

# Replay Protection

Requests using a capability token must include replay protection.

Loopgate must track request nonces or equivalent replay identifiers and reject
reuse deterministically.

---

# Token Revocation

Loopgate must support token revocation.

Revocation triggers may include:

- user cancellation
- morphling termination
- policy update
- security anomaly

Revoked tokens must immediately become invalid.

---

# Token Storage

Tokens must not be stored persistently by Morphlings.

Allowed storage:

- in-memory during execution

Disallowed storage:

- disk persistence
- inclusion in artifacts

Loopgate must control token lifecycle.

---

# Audit Requirements

Loopgate must record the following events:

- token issuance
- token validation failure
- token expiration
- token revocation
- token usage

These events must append to the audit ledger.

---

# Security Invariants

1. Only Loopgate may issue tokens.
2. Tokens must always be scoped.
3. Tokens must expire.
4. Request integrity must be independently verified.
5. Operator clients and morphlings must never self-issue authority.
4. Tokens must be signed.
5. Tokens must bind to a specific task.
6. Tokens must never grant root-level filesystem access.
7. Tokens must never bypass policy checks.

---

# Future Work

Possible improvements:

- hierarchical token scopes
- distributed token verification
- hardware-backed signing
- zero-trust token rotation

These are **out of scope for v1**.

---

# Conclusion

Capability tokens are the mechanism that enforces **authority boundaries** between operator clients, morphlings, and the host system.

By requiring all privileged operations to pass through Loopgate-issued tokens, the system maintains strong guarantees about scope, safety, and auditability.
